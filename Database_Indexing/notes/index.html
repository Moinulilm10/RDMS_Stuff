Stage 4: Database Indexing

Mental Model:
An index is exactly like the index at the back of a textbook. It's a sorted copy of specific columns pointing back to the physical page numbers (Row IDs).

1. The Write Tax (Crucial!)

Before making an index, you must understand The Write Tax. Indexes speed up reads, but they severely slow down writes!

If a table has 10 indexes, a single INSERT into the table requires 11 physical disk writes! (1 write for the heap table, and 10 writes to update the B+ Tree for every single index). Never over-index!

2. Advanced Index Structures

While B+ Trees are standard, modern databases (like PostgreSQL) offer specialized structures for complex data:

  • GIN (Generalized Inverted Index): Used for JSONB, Arrays, and Full-Text Search. Instead of indexing a whole document, it indexes every single word/key and points to a list of rows containing that element.
  • GiST (Generalized Search Tree): Used for geospatial shapes (PostGIS). It operates on overlapping bounding boxes. If two shapes overlap, GiST helps find candidates fast!
  • BRIN (Block Range Index): Used for massive, naturally sorted time-series tables (e.g., IoT logs). Instead of indexing every single row, it stores the MIN/MAX timestamp for every 1MB physical block on disk. This index is tiny in RAM but allows skipping massive amounts of irrelevant disk blocks.

3. Reading Execution Plans

The Cost-Based Optimizer (CBO) decides if your index is actually worth using by calculating Selectivity (how many rows will match the filter).

EXPLAIN ANALYZE SELECT * FROM users WHERE status = 'active';
  • Index Scan: The CBO traverses the B+ Tree, finds the row pointer, and fetches the row from the heap. Great for high-selectivity (rare) queries!
  • Seq Scan (Full Table Scan): The CBO IGNORES your index! If the query returns 30% of the table, jumping back and forth between the index and the heap (Random I/O) is slower than just reading the entire table sequentially from start to finish!
Bitmap Index Scans:
If you have two indexes (e.g., age, city) and a query with an AND clause, Postgres scans both indexes, builds an in-memory bit-array for each, performs a fast bitwise AND in RAM, and then fetches the matching rows from disk sequentially!

4. Covering & Partial Indexes

If you have an API endpoint that only needs a user's status: SELECT status FROM users WHERE id = 5;

You can create a Covering Index: CREATE INDEX idx_user ON users(id) INCLUDE (status);. The database never has to touch the physical table! It answers the query entirely from the Index Leaf Node (an Index-Only Scan).

A Partial Index saves space by only indexing a subset: CREATE INDEX idx_pending ON tasks(date) WHERE state = 'pending';. This index is tiny, fast, and stays cached in RAM!
