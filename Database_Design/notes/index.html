Module 1: ERD & Relational Entities

An Entity Relationship Diagram (ERD) is the blueprint translating business domain rules into storage structures before writing SQL schema code.

1.1 One-to-One (1:1) Relationships

🛑 Real-World Problem Solved: Table Bloat & Security Exposure If you put 50 columns inside `users`—including password hashes, Social Security Numbers (SSNs), and 10MB user bio markdown files—every single login check or profile query loads bloated data into memory and risks leaking PII to backend queries.
✅ Architectural Solution: Vertical Partitioning Isolate heavy or highly classified PII attributes into a separate child table linked by a Primary Key that doubles as a Foreign Key (`user_id`).
SQL Implementation: 1:1 KYC Security Isolation CREATE TABLE users ( user_id INTEGER PRIMARY KEY AUTOINCREMENT, email VARCHAR(255) NOT NULL UNIQUE, account_status VARCHAR(32) DEFAULT 'ACTIVE' ); -- Child table shares identical user_id PK/FK: CREATE TABLE user_kyc_pii ( user_id INTEGER PRIMARY KEY, ssn_encrypted VARCHAR(512) NOT NULL UNIQUE, tax_id VARCHAR(64) NOT NULL, FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE );

1.2 One-to-Many (1:M) Relationships

🛑 Real-World Problem Solved: Repeating Data Duplication In E-commerce, if one order has 10 line items, storing everything in one flat table repeats the customer's name, email, and shipping address 10 times!

In 1:M architecture, the Foreign Key column ALWAYS resides on the Many (<) child side pointing to the parent's unique ID.

SQL Implementation: 1:M Order Line Items CREATE TABLE orders ( order_id INTEGER PRIMARY KEY AUTOINCREMENT, customer_email VARCHAR(255) NOT NULL, created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP ); CREATE TABLE order_items ( item_id INTEGER PRIMARY KEY AUTOINCREMENT, order_id INTEGER NOT NULL, -- FK sits on Many side sku VARCHAR(64) NOT NULL, quantity INTEGER NOT NULL CHECK (quantity > 0), FOREIGN KEY (order_id) REFERENCES orders(order_id) ON DELETE CASCADE );

1.3 Many-to-Many (M:N) & Junction Payloads

🛑 Real-World Problem Solved: Infinite Loops & Multiple Enrollment Students enroll in many courses, and courses have many students. You cannot put a Foreign Key in either table without creating infinite circular dependency loops!
✅ Architectural Solution: Junction (Association) Table Create a bridging table with a composite primary key `(student_id, course_id)`. Crucially, junction tables store Relationship Payloads like enrollment date or final grade!
SQL Implementation: M:N Junction Payload CREATE TABLE enrollments ( student_id INTEGER NOT NULL, course_id INTEGER NOT NULL, semester_enrolled VARCHAR(16) NOT NULL, -- Payload attribute final_grade VARCHAR(4), -- Payload attribute PRIMARY KEY (student_id, course_id), -- Composite PK prevents double enrollment! FOREIGN KEY (student_id) REFERENCES students(student_id), FOREIGN KEY (course_id) REFERENCES courses(course_id) );

Module 2: Key Architecture & Primary Key Engineering

2.1 Natural Keys vs Surrogate Keys

🛑 Real-World Problem Solved: The Enterprise Mutability Crisis If you use a natural attribute like employee email or company tax ID as the Primary Key, what happens when the company rebrands or an employee changes their legal name? Updating a primary key forces taking global table locks to update millions of referencing foreign keys!

Always use an immutable Surrogate Key (`user_id INTEGER / UUID`) as the Primary Key, and protect natural domain keys with a `UNIQUE` constraint!

2.2 Distributed ID Generation at Scale

🛑 Real-World Problem Solved: Multi-Master Auto-Increment Bottlenecks In globally sharded databases (Instagram, Uber), sequential auto-increment (`1, 2, 3...`) requires centralized global locks across data centers that throttle write throughput to near zero.
Strategy Pros & Storage Cons & Best Used For
Auto-Increment INT Compact 4 or 8 bytes. Sequential B-Tree inserts. Single point of failure. Cannot generate IDs offline on mobile apps.
UUID v4 (Random) 128-bit string. Offline generation without central server. Random entropy destroys B-Tree locality, causing massive page splits & 50% index bloat.
UUID v7 / ULID Stores Unix Timestamp in leading bits + random entropy. ⭐ Industry Standard! Offline generation AND time-ordered sequential B-Tree appends!
Snowflake ID 64-bit integer (Timestamp + Worker ID + Counter). Compact 8-byte integer used by Twitter/Instagram sharded systems.
SQL Implementation: UUIDv7 Time-Ordered Event Streams CREATE TABLE telemetry_events ( event_id UUID PRIMARY KEY, -- Storing time-ordered UUIDv7 worker_node_id INTEGER NOT NULL, payload JSONB NOT NULL, created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP );

Module 3: Domain Integrity & Table Constraints

3.1 CHECK & UNIQUE Constraints

🛑 Real-World Problem Solved: Concurrent Race Conditions If two payment requests check account balance simultaneously (`SELECT balance FROM wallet`), both see $100 and approve $80 withdrawals. Without storage engine checks, accounts drop to -$60!

A CHECK constraint executes atomically at the storage layer during row locking, rejecting illegal mathematical transitions!

SQL Implementation: Stripe Wallet Mathematical Guarantees CREATE TABLE customer_wallets ( wallet_id UUID PRIMARY KEY, customer_id UUID NOT NULL UNIQUE, balance DECIMAL(15, 4) NOT NULL DEFAULT 0.0000 CHECK (balance >= 0.0000), currency VARCHAR(3) NOT NULL DEFAULT 'USD', status VARCHAR(16) NOT NULL CHECK (status IN ('ACTIVE', 'FROZEN', 'CLOSED')) );

3.2 Foreign Key Referential Actions

🛑 Real-World Problem Solved: Orphaned Rows & Audit Loss If a department manager is deleted from HR systems, what happens to their 50 subordinates?
  • ON DELETE RESTRICT: Blocks deletion if child rows exist. Mandatory for financial ledgers!
  • ON DELETE CASCADE: Deletes child rows automatically. Perfect for user shopping carts.
  • ON DELETE SET NULL: Preserves child records by nullifying the FK. Perfect for retaining employee records when a department dissolves.

Module 4: Normalization Mastery (1NF to 5NF)

Normalization systematically removes data anomalies across enterprise tables:

4.1 First Normal Form (1NF): Atomicity Rules

🛑 Real-World Problem Solved: Un-Indexable CSV Strings Storing comma-separated tags like `"java, aws, sql"` forces slow full table scans (`WHERE skills LIKE '%sql%'`).
SQL Implementation: Decomposing 1NF Atomic Table CREATE TABLE developer_skills_1nf ( developer_id INTEGER NOT NULL, skill_name VARCHAR(64) NOT NULL, years_experience INTEGER NOT NULL, PRIMARY KEY (developer_id, skill_name) -- Composite PK guarantees atomicity! );

4.2 Second Normal Form (2NF): Partial Dependencies

🛑 Real-World Problem Solved: Price Corruption on Historical Orders If order line items store `(order_id, product_sku, product_name, current_price)`, updating a product's price overwrites historical order receipts from 3 years ago!

2NF requires that non-key attributes depend on the entire composite primary key.

4.3 Third Normal Form (3NF) & BCNF: Transitive Dependencies

🛑 Real-World Problem Solved: Transitive Data Corruption If an employee row contains `(emp_id, dept_id, building_name, helpdesk_phone)`, notice helpdesk phone depends on building name, which depends on dept_id!

3NF rule: Non-key attributes must depend directly on the primary key, the whole key, and nothing but the key!

4.4 Fourth Normal Form (4NF): Multi-Valued Dependencies (MVDs)

🛑 Real-World Problem Solved: Cartesian Explosion If a consultant speaks 3 languages (English, Spanish, German) and holds 2 certifications (AWS, CISSP), combining them into one table creates 3 × 2 = 6 redundant Cartesian product rows!
SQL Implementation: 4NF Independent Dimensions CREATE TABLE consultant_languages ( consultant_id INTEGER NOT NULL, language VARCHAR(32) NOT NULL, PRIMARY KEY (consultant_id, language) ); CREATE TABLE consultant_certifications ( consultant_id INTEGER NOT NULL, cert_name VARCHAR(64) NOT NULL, PRIMARY KEY (consultant_id, cert_name) );

Module 5: Denormalization & Analytical OLAP

5.1 Why Denormalize? Read-Heavy Caching

🛑 Real-World Problem Solved: High-Traffic N+1 CPU Starvation Running `SELECT COUNT(*) FROM comments WHERE post_id = ?` across 100 million Reddit comments on every viral post view causes database CPU exhaustion.
✅ Architectural Solution: Pre-Aggregated Counter Triggers Store a denormalized `cached_comment_count` directly on the parent post record, maintained automatically via database triggers.
SQL Implementation: Denormalized Counter Cache Trigger CREATE TABLE posts ( post_id INTEGER PRIMARY KEY, title VARCHAR(255) NOT NULL, cached_comment_count INTEGER DEFAULT 0 ); CREATE TRIGGER trg_increment_comment_count AFTER INSERT ON comments BEGIN UPDATE posts SET cached_comment_count = cached_comment_count + 1 WHERE post_id = NEW.post_id; END;

5.2 Dimensional Modeling: Star vs Snowflake Schemas

In Data Warehouses (BigQuery, Snowflake), business intelligence queries aggregate billions of rows. We structure schemas into central Fact Tables surrounded by unnormalized descriptive Dimension Tables.

Module 6: Migrations & DDL Lifecycle

🛑 Real-World Problem Solved: Manual Production DDL Outages Executing manual `ALTER TABLE` scripts in production terminal prompts risks locking high-concurrency tables and crashing live microservices.

Always use schema migration frameworks (Flyway, Liquibase, Alembic) executing immutable, numbered UP/DOWN scripts in CI/CD pipelines!

Flyway Migration: V20260702__add_user_tier.sql -- UP Migration Script: ALTER TABLE user_accounts ADD COLUMN subscription_tier VARCHAR(32) NOT NULL DEFAULT 'FREE'; -- DOWN Rollback Script: ALTER TABLE user_accounts DROP COLUMN subscription_tier;

Module 7: Advanced Enterprise Design Patterns

7.1 Soft Deletes & Filtered Active Indexes

🛑 Real-World Problem Solved: Audit Destruction & Broken Analytics Executing physical `DELETE FROM orders` destroys financial revenue trends and violates tax accounting retention compliance.

Instead, add a nullable timestamp `deleted_at TIMESTAMPTZ NULL`. To keep queries blazing fast over millions of soft-deleted records, build Partial Filtered Indexes!

SQL Implementation: Soft Deletes & Partial Indexing CREATE TABLE customer_contracts ( contract_id UUID PRIMARY KEY, company_name VARCHAR(128) NOT NULL, annual_value DECIMAL(12,2) NOT NULL, deleted_at TIMESTAMPTZ NULL -- NULL means active! ); -- Partial index indexes ONLY active records, saving 80% disk space! CREATE INDEX idx_active_contracts ON customer_contracts(company_name) WHERE deleted_at IS NULL;

7.2 Multi-Tenancy SaaS Architecture

How B2B SaaS platforms (Salesforce, Slack) isolate client data across 10,000 corporate organizations:

  • Shared Schema (Pool): All tables include `tenant_id UUID`. Enforced via Row-Level Security (RLS). Maximum scalability!
  • Separate Schema: Dedicated schema per client (`tenant_acme.users`).
  • Separate DB Instance (Silo): Dedicated physical DB server per enterprise client (Mandatory for HIPAA / SOC2 defense contracts).
SQL Implementation: PostgreSQL Row-Level Security (RLS) ALTER TABLE tenant_messages ENABLE ROW LEVEL SECURITY; CREATE POLICY tenant_isolation_policy ON tenant_messages USING (tenant_id = current_setting('app.current_tenant_id')::uuid);
