📚 Chapter 1: Database Basics

What is a Database?

A database is an organized collection of structured data, stored and accessed electronically. Think of it as a super-powered spreadsheet that can handle millions of records!

RDBMS (Relational Database Management System)

An RDBMS stores data in tables (rows & columns), connected by relationships (foreign keys).

⭐ Key Concept

Popular RDBMS: PostgreSQL 🐘, MySQL 🐬, SQL Server, Oracle, SQLite

Key Feature: Data is organized into tables with rows (records) and columns (attributes)

SQL vs NoSQL

FeatureSQL (Relational)NoSQL
SchemaFixed, predefinedDynamic, flexible
ScalingVertical (bigger server)Horizontal (more servers)
TransactionsFull ACID supportEventually consistent
JoinsNative JOIN supportManual / Denormalized
Best ForComplex queries, relationshipsFlexible data, high throughput
💡 Rule of Thumb: If your data has relationships and you need consistency → SQL. If you need flexibility and massive scale → NoSQL. Many systems use BOTH!

Architecture Hierarchy

Server (Hardware/VM) └── Instance (Running DB process) ├── Database: ecommerce │ ├── Schema: public │ │ ├── Table: users ← rows & columns │ │ ├── Table: orders │ │ └── Table: products │ └── Schema: analytics └── Database: blog

Key Terms

  • Table — Collection of related data in rows & columns
  • Row (Tuple) — A single record (one employee, one order)
  • Column (Attribute) — A specific field (name, salary, email)
  • Schema — Logical namespace grouping related objects
  • Metadata — Data about data (column types, indexes, stats)

🔒 Chapter 3: Constraints

Types of Constraints

🔑 PRIMARY KEY

Uniquely identifies each row. Cannot be NULL. Must be unique. Every table needs one!

CREATE TABLE employees ( id INTEGER PRIMARY KEY AUTOINCREMENT, email TEXT UNIQUE NOT NULL ); -- Composite PK: CREATE TABLE enrollment ( student_id INT, course_id INT, PRIMARY KEY (student_id, course_id) );

🔗 FOREIGN KEY

Links two tables. Enforces referential integrity — FK value must exist in the parent table.

ON DELETE ActionBehavior
CASCADEDelete child rows automatically
SET NULLSet FK to NULL in children
RESTRICTPrevent deletion if children exist
SET DEFAULTSet FK to default value

Other Constraints

  • NOT NULL — Column must have a value
  • UNIQUE — No duplicate values in column
  • CHECK — Custom validation: CHECK(price > 0)
  • DEFAULT — Auto-fill: DEFAULT 'draft'
Best Practice: Always use NOT NULL unless you have a specific reason for NULLs. Use CHECK constraints to enforce business rules at the database level!

✏️ Chapter 4: CRUD Operations

INSERT — Create

-- Single row: INSERT INTO users (name, email) VALUES ('Alice', 'alice@test.com'); -- Batch (MUCH faster!): INSERT INTO users (name, email) VALUES ('Bob', 'bob@test.com'), ('Carol', 'carol@test.com'); -- From SELECT: INSERT INTO archive SELECT * FROM orders WHERE status = 'done';

SELECT — Read

SELECT first_name, salary FROM employees WHERE salary > 50000; SELECT DISTINCT department_id FROM employees; -- No duplicates SELECT *, salary * 12 AS annual FROM employees; -- Computed column
⭐ Execution Order

SQL does NOT execute in the order you write it!

FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

This is why you CAN'T use column aliases in WHERE (SELECT runs after WHERE!)

UPDATE — Modify

UPDATE employees SET salary = salary * 1.10 WHERE department_id = 1;

DELETE — Remove

DELETE FROM orders WHERE order_date < '2020-01-01';
🚨 DANGER

Always use WHERE with UPDATE and DELETE!

UPDATE employees SET salary = 0; — Updates EVERYONE!

DELETE FROM employees; — Deletes EVERYTHING!

Test with SELECT first: SELECT * FROM employees WHERE condition;

DELETE vs TRUNCATE vs DROP

CommandRemovesRollback?Speed
DELETESpecific rows✅ YesSlow (row by row)
TRUNCATEAll rows❌ NoVery fast
DROPEntire table❌ NoInstant

🔍 Chapter 5: Filtering (WHERE)

Comparison Operators

= equal, != or <> not equal, > greater, >= greater/equal, < less, <= less/equal

LIKE — Pattern Matching

WHERE name LIKE 'A%' -- Starts with A WHERE email LIKE '%@gmail.com' -- Ends with @gmail.com WHERE name LIKE '_a%' -- 2nd letter is 'a' -- % = any characters, _ = exactly one character
⚡ Performance: LIKE '%keyword' (leading %) can't use indexes → full table scan! Use LIKE 'keyword%' when possible.

IN & BETWEEN

-- IN: Set membership WHERE dept_id IN (1, 3, 5) -- Same as dept_id=1 OR dept_id=3 OR dept_id=5 -- BETWEEN: Range (inclusive on BOTH ends!) WHERE salary BETWEEN 50000 AND 80000 -- Same as: salary >= 50000 AND salary <= 80000

EXISTS & NOT EXISTS

-- EXISTS: Returns TRUE if subquery returns ANY rows SELECT d.name FROM departments d WHERE EXISTS ( SELECT 1 FROM employees e WHERE e.department_id = d.id AND e.salary > 80000 );
🚨 NOT IN Trap

WHERE x NOT IN (1, 2, NULL) returns NO ROWS!

Because x != NULL is always UNKNOWN. Use NOT EXISTS instead!

EXISTS vs IN

  • EXISTS stops at first match (short-circuits) → faster for large subqueries
  • IN evaluates entire subquery first → better for small static lists
  • EXISTS handles NULLs correctly; NOT IN fails with NULLs!

📊 Chapter 6: Sorting & Pagination

-- ORDER BY: SELECT * FROM employees ORDER BY salary DESC; SELECT * FROM employees ORDER BY dept_id ASC, salary DESC; -- LIMIT & OFFSET (Pagination): SELECT * FROM employees ORDER BY id LIMIT 10; -- Page 1 SELECT * FROM employees ORDER BY id LIMIT 10 OFFSET 10; -- Page 2 -- Formula: LIMIT size OFFSET (page - 1) * size
⚠️ OFFSET Problem

OFFSET 1000000 still scans 1 million rows → very slow!

Better: Keyset pagination: WHERE id > last_seen_id ORDER BY id LIMIT 10

📦 Chapter 8: GROUP BY & HAVING

GROUP BY

SELECT department_id, COUNT(*), AVG(salary) FROM employees GROUP BY department_id;
⭐ Golden Rule

Every column in SELECT must either be in GROUP BY or inside an aggregate function!

WHERE vs HAVING

FeatureWHEREHAVING
FiltersIndividual rowsGroups (after GROUP BY)
RunsBEFORE GROUP BYAFTER GROUP BY
Aggregates?❌ Cannot use✅ Can use

❓ Chapter 9: NULL Handling

Three-Valued Logic

SQL has three values: TRUE, FALSE, and UNKNOWN. Any comparison with NULL = UNKNOWN.

ExpressionResult
NULL = NULLUNKNOWN ❌ (not TRUE!)
NULL IS NULLTRUE ✅
NULL + 10NULL (any math with NULL = NULL)
🚨 Never use = NULL!

❌ WHERE col = NULL → returns NO rows!

✅ WHERE col IS NULL → correct!

COALESCE & NULLIF

-- COALESCE: Returns first non-NULL value COALESCE(nickname, first_name, 'Anonymous') -- NULLIF: Returns NULL if both args are equal -- Key use: Prevent division by zero! total / NULLIF(count, 0) -- Returns NULL instead of error
Portability Tip: Always use COALESCE (SQL standard) instead of IFNULL (MySQL/SQLite), NVL (Oracle), or ISNULL (SQL Server).

🔗 Chapter 10: Joins

Join Types Visual Guide

INNER JOIN: Only matching rows (intersection) A ∩ B → [matched rows only] LEFT JOIN: All from A + matched from B A ← B → [all A + matching B, NULLs for unmatched] RIGHT JOIN: All from B + matched from A A → B → [all B + matching A, NULLs for unmatched] FULL JOIN: All from both tables A ↔ B → [everything, NULLs where no match] CROSS JOIN: Every combination (Cartesian product) A × B → [n × m rows!] SELF JOIN: Table joined with itself A ⟳ A → [hierarchies, comparisons]

INNER JOIN

SELECT e.name, d.name AS dept FROM employees e INNER JOIN departments d ON e.dept_id = d.id;

Most common join (~80%) — returns only matching rows from both tables.

LEFT JOIN

-- All employees (even without department): SELECT e.name, d.name AS dept FROM employees e LEFT JOIN departments d ON e.dept_id = d.id; -- Anti-join pattern (find orphans): SELECT e.name FROM employees e LEFT JOIN departments d ON e.dept_id = d.id WHERE d.id IS NULL; -- Employees with NO department

SELF JOIN

-- Employee → Manager hierarchy: SELECT e.name AS employee, m.name AS manager FROM employees e LEFT JOIN employees m ON e.manager_id = m.id;
🚨 LEFT JOIN + WHERE Trap

LEFT JOIN d ON ... WHERE d.location = 'NYC'

This turns LEFT JOIN into INNER JOIN! The WHERE filters out NULLs.

✅ Put condition in ON clause instead: ON ... AND d.location = 'NYC'

How Joins Work Internally:
🔄 Nested Loop — O(n×m), small tables
#️⃣ Hash Join — O(n+m), large unsorted tables
⬇️ Merge Join — O(n log n), sorted/indexed data

🎯 Chapter 12: Subqueries

Non-Correlated (Independent)

Executes once, independently of the outer query.

-- Employees above average salary: SELECT * FROM employees WHERE salary > (SELECT AVG(salary) FROM employees);

Correlated (Dependent)

Executes once per outer row — references the outer query.

-- Employees above THEIR department's average: SELECT * FROM employees e WHERE salary > ( SELECT AVG(salary) FROM employees e2 WHERE e2.dept_id = e.dept_id -- References outer! );
⚠️ Performance

Correlated subqueries run once per outer row. 10,000 rows = 10,000 executions!

Often a JOIN or window function is faster.

🏗️ Chapter 13: CTEs (Common Table Expressions)

Basic CTE

WITH dept_stats AS ( SELECT dept_id, AVG(salary) AS avg_sal FROM employees GROUP BY dept_id ) SELECT e.name, d.avg_sal FROM employees e JOIN dept_stats d ON e.dept_id = d.dept_id;

Recursive CTE

WITH RECURSIVE org_tree AS ( -- Base: Start from CEO SELECT id, name, manager_id, 0 AS level FROM employees WHERE manager_id IS NULL UNION ALL -- Recursive: Find direct reports SELECT e.id, e.name, e.manager_id, ot.level + 1 FROM employees e JOIN org_tree ot ON e.manager_id = ot.id ) SELECT * FROM org_tree ORDER BY level;
CTE vs Subquery vs Temp Table:
📖 CTE — Best readability, reusable within query, supports recursion
🔄 Subquery — Can't reuse, hard to read when nested
💾 Temp Table — Materialized on disk, reusable across queries

🪟 Chapter 14: Window Functions

Syntax

function() OVER ( PARTITION BY col -- Groups (like GROUP BY but keeps all rows!) ORDER BY col -- Order within partition ROWS BETWEEN ... -- Frame: which rows to include )
⭐ Key Difference from GROUP BY

GROUP BY: Collapses rows → one row per group

Window Functions: Keeps ALL rows, adds computed column

Ranking Functions

Salary: 100K 90K 90K 80K 70K ROW_NUMBER: 1 2 3 4 5 ← Always unique RANK: 1 2 2 4 5 ← Gaps after ties DENSE_RANK: 1 2 2 3 4 ← No gaps!

Top-N Per Group Pattern ⭐

-- Top 2 earners per department: SELECT * FROM ( SELECT *, ROW_NUMBER() OVER ( PARTITION BY dept_id ORDER BY salary DESC ) AS rn FROM employees ) WHERE rn <= 2;

LAG & LEAD

LAG(col, n) — value from n rows BEFORE current

LEAD(col, n) — value from n rows AFTER current

-- Month-over-month growth: SELECT month, revenue, revenue - LAG(revenue) OVER (ORDER BY month) AS growth FROM monthly_sales;

Window Frames

-- Running total: SUM(amt) OVER (ORDER BY date ROWS UNBOUNDED PRECEDING) -- Moving average (last 3): AVG(price) OVER (ORDER BY date ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) -- Percentage of total: salary / SUM(salary) OVER () * 100
🚨 LAST_VALUE Trap

Default frame = ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW

So LAST_VALUE just returns current row! Always specify full frame for LAST_VALUE:

ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING

🔀 Chapter 15: CASE Expression

-- Searched CASE (any condition): CASE WHEN salary >= 100000 THEN 'Executive' WHEN salary >= 70000 THEN 'Senior' ELSE 'Junior' END -- Simple CASE (equality): CASE dept_id WHEN 1 THEN 'Eng' WHEN 2 THEN 'Mkt' ELSE 'Other' END -- In ORDER BY (custom sort!): ORDER BY CASE status WHEN 'urgent' THEN 1 WHEN 'normal' THEN 2 ELSE 3 END -- Conditional aggregation (pivot!): SELECT COUNT(CASE WHEN status='active' THEN 1 END) AS active_count
CASE evaluates top-down! Put specific conditions before general ones. Returns first match and stops.

⚡ Chapter 17: Indexes

An index is a B-tree data structure that speeds up lookups from O(n) to O(log n).

CREATE INDEX idx_emp_dept ON employees(department_id); CREATE UNIQUE INDEX idx_emp_email ON employees(email); CREATE INDEX idx_composite ON employees(dept_id, salary); -- Multi-column

When to Index ✅

  • Columns in WHERE clauses
  • Foreign key columns
  • ORDER BY columns
  • High-cardinality columns

When NOT to Index ❌

  • Small tables (< 1000 rows)
  • Low-cardinality (boolean, gender)
  • Heavy INSERT/UPDATE tables (index maintenance overhead)
  • Rarely queried columns

⚙️ Chapter 19: Stored Procedures

Pre-compiled SQL code stored in the database. Accepts parameters, contains logic (IF/ELSE, loops).

-- PostgreSQL: CREATE FUNCTION give_raise(emp_id INT, pct DECIMAL) RETURNS VOID AS $$ BEGIN UPDATE employees SET salary = salary * (1 + pct/100) WHERE id = emp_id; END; $$ LANGUAGE plpgsql;
  • Pros: Reduced network traffic, centralized logic, security, pre-compiled
  • Cons: Vendor lock-in, hard to debug, version control issues, scaling challenges

🎯 Chapter 20: Triggers

Special procedures that automatically execute on INSERT, UPDATE, or DELETE events.

CREATE TRIGGER log_salary_change AFTER UPDATE OF salary ON employees FOR EACH ROW BEGIN INSERT INTO audit_log (action, old_val, new_val) VALUES ('SALARY', OLD.salary, NEW.salary); END;
TimingWhenUse Case
BEFOREBefore the operationValidation, auto-fill
AFTERAfter the operationAudit logging
INSTEAD OFReplaces the operationUpdatable views
🚨 Trigger Dangers

Triggers are "invisible" — they run without explicit calls. Cascading triggers = debugging nightmares. Use sparingly!

📐 Chapter 21: Normalization

1NF — Atomic Values

Each cell has one value only. No comma-separated lists! Each row is unique (has PK).

2NF — No Partial Dependencies

Non-key columns depend on the ENTIRE primary key, not just part of it.

3NF — No Transitive Dependencies

Non-key columns can't depend on other non-key columns.

❌ emp_id → zip_code → city (city depends on zip, not emp_id) ✅ employees(emp_id, zip_code) + locations(zip_code, city)
Denormalization: Sometimes you intentionally break normalization for performance. Read-heavy dashboards benefit from pre-joined data. Trade-off: faster reads vs slower writes + potential inconsistency.

📦 Chapter 23: Partitioning & Sharding

Table Partitioning

Splitting a massive physical table into smaller table files across disk while keeping a single logical table interface!

MethodHow It Splits DataBest Use Case
RangeContinuous intervals (dates, IDs)Time-series logs (`orders_2026_01`)
ListSpecific categorical valuesRegions (`users_usa`, `users_eu`)
HashModulo hash of partition keyEven write balancing across N disks
⭐ Partition Pruning

When query filter matches partition key (`WHERE date >= '2026-06-01'`), the DB skips scanning irrelevant disk partitions completely! Massive speedup!

Sharding (Horizontal Partitioning across Servers)

When vertical scaling (bigger RAM/CPU) hits limits, data is split across multiple physical database servers (`Shard A`, `Shard B`). Requires router/routing logic!

⚡ Chapter 24: Concurrency & MVCC

Multi-Version Concurrency Control (MVCC)

How databases allow thousands of users to read and write without locking each other out:

⭐ The Golden Rule of MVCC: Readers NEVER block Writers, and Writers NEVER block Readers!

How MVCC Works Internally

  • Each row has hidden timestamp/version columns (`xmin`, `xmax`)
  • Updates don't overwrite rows; they write a new row version!
  • Readers see a snapshot of rows valid at their transaction start time
  • Dead old versions cleaned up by background garbage collection (`VACUUM`)

Locking Strategies

-- Pessimistic Locking (lock row during read): SELECT * FROM accounts WHERE id = 1 FOR UPDATE; -- Optimistic Locking (check version number before save): UPDATE accounts SET balance=100, version=version+1 WHERE id=1 AND version=5; -- Fails if someone updated it!

📄 Chapter 25: JSON & NoSQL in SQL

Why Hybrid SQL + JSON?

Combine rigid relational schemas for core entities (Users, Orders) with flexible JSON documents for dynamic metadata (Settings, Attributes)!

-- PostgreSQL JSONB Queries: -- Extract text property (->>): SELECT metadata->>'theme' FROM users WHERE metadata->>'tier' = 'pro'; -- Check array containment (@>): SELECT * FROM products WHERE specs @> '{"wireless": true}'; -- Inverted GIN Index (blazing fast JSON queries!): CREATE INDEX idx_prod_specs ON products USING GIN (specs);
💡 JSON vs JSONB in PostgreSQL: `JSON` stores exact raw text (slower queries). `JSONB` stores parsed binary representation (slightly slower insert, 10x faster queries + indexing!). Always prefer `JSONB`!

🛡️ Chapter 26: Security & Row-Level Security

DCL (Data Control Language)

GRANT SELECT, INSERT ON orders TO app_user; REVOKE DELETE ON orders FROM app_user;

Row-Level Security (RLS) ⭐

Enforces security filters at the engine level so users can ONLY see their own tenant rows, even with `SELECT *`!

ALTER TABLE tenant_orders ENABLE ROW LEVEL SECURITY; CREATE POLICY tenant_isolation ON tenant_orders USING (tenant_id = current_setting('app.tenant_id')::uuid);

Prevents multi-tenant SaaS data leaks at the database layer!

💾 Chapter 27: Backup & Replication

Write-Ahead Logging (WAL)

Every change is sequentially appended to a WAL log on disk before modifying database table pages. Ensures durability + instant crash recovery!

Point-In-Time Recovery (PITR)

By combining weekly full disk backups with continuous stream WAL logs, you can restore your database to any exact second in time (e.g., 5 seconds before someone ran `DROP TABLE`)!

Replication Architectures

  • Primary-Replica (Master-Slave): All writes go to Primary; Replicas handle read-heavy queries.
  • Synchronous vs Async: Sync waits for replica confirmation (zero data loss); Async responds immediately (faster).
