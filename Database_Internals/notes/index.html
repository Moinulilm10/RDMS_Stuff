⚙️ Database Internals & Storage Engines

Stage 3 ⭐⭐⭐⭐⭐ · What Separates Juniors from Seniors · Print PDF Ready
💡 The Senior Engineer Mindset: Never treat a database as a black box! When a system scales to 50,000 requests/sec, your ability to diagnose disk I/O, buffer pool thrashing, B+ tree page splits, and WAL synchronization is what defines Staff-level engineering!

Module 1: The Lifecycle of a SQL Query

What actually happens under the hood when you execute SELECT * FROM users WHERE id = 42;?

🛑 Real-World Problem Solved: Blind Performance Bottlenecks When a query times out under heavy traffic, juniors randomly add indexes or reboot instances. Seniors trace the exact 5-stage engine pipeline to pinpoint whether the bottleneck is parsing CPU, stale statistics in the planner, or random disk I/O in the storage engine!

The 5-Stage Execution Pipeline

  1. 1. SQL Parser: Tokenizes the SQL string and builds an Abstract Syntax Tree (AST). Validates grammar and syntax rules.
  2. 2. Analyzer / Binder: Checks the AST against the System Catalog (data dictionary) to verify table names, column types, and IAM access permissions.
  3. 3. Query Planner & CBO: The Cost-Based Optimizer (CBO) evaluates multiple execution paths using statistical histograms (cardinality and selectivity) to pick the cheapest disk I/O plan!
  4. 4. Execution Engine: Pulls tuples using the Volcano Iterator Model (calling open(), next(), and close() on tree operators).
  5. 5. Storage Engine: Checks if requested 4KB/8KB pages reside in the in-memory Buffer Pool. If not, fetches physical blocks from SSD/disk!
SQL Pro-Tip: Bypassing Parser & Planner Overhead -- In high-throughput APIs (e.g., Uber GPS ingestion or Stripe checkout), -- parsing SQL on every request wastes up to 30% of database CPU! -- Use PREPARED STATEMENTS to compile and cache the execution plan once: PREPARE get_user (int) AS SELECT id, email, status FROM users WHERE id = $1; -- Execute cached plan instantly without re-parsing: EXECUTE get_user(42);

Module 2: Memory Architecture & The Buffer Pool

2.1 Why Buffer Pools Exist: RAM vs Disk Latency

Reading from physical disk (even NVMe SSDs) takes ~100,000 nanoseconds. Reading from RAM takes ~100 nanoseconds. The Buffer Pool is an in-memory cache managed directly by the database storage engine, dividing RAM into fixed 4KB, 8KB, or 16KB memory frames that mirror physical disk blocks!

🛑 Real-World Problem Solved: Cache Pollution & Thrashing Under basic LRU (Least Recently Used), if a data analyst runs an analytical dump (SELECT * FROM old_audit_logs scanning 500GB of cold data), every hot production page is evicted from RAM! The database freezes for all live customers!
✅ Architectural Solution: Scan-Resistant Eviction (Clock Sweep / LRU-K) Modern engines use Midpoint Insertion (InnoDB) or 2Q / LRU-K (PostgreSQL). New pages read from disk enter at the midpoint! They only get promoted to the hot working set if they are accessed a *second* time within a specific time window!
Monitoring Buffer Pool Hit Ratio in PostgreSQL -- If Hit Ratio drops below 99.0%, RAM is under-provisioned or queries are doing table scans! SELECT sum(heap_blks_read) as disk_reads, sum(heap_blks_hit) as buffer_hits, round(sum(heap_blks_hit) / (sum(heap_blks_hit) + sum(heap_blks_read))::numeric * 100, 2) as hit_ratio_percentage FROM pg_statio_user_tables;

Module 3: Disk Storage & Page Anatomy

3.1 Slotted Page Architecture

Why don't databases store rows contiguously like text files? Because variable-length strings (VARCHAR(255)) would leave fragmented holes whenever a row is deleted or updated!

All major RDBMS (PostgreSQL, MySQL/InnoDB, SQLite) use Slotted Page Architecture inside each fixed 4KB/8KB disk block:

  • Line Pointers (Slot Array): Fixed-size pointers at the top of the page growing DOWNWARDS. An index references a tuple via its Row ID (RID = Page# + Slot#).
  • Tuple Data: Actual row data inserted at the bottom of the page growing UPWARDS!
  • Free Space: The empty space in the middle where Line Pointers and Tuples meet!
💡 Why Slotted Pages are Brilliant: When a tuple moves within a page during compaction, ONLY the internal offset inside the Line Pointer changes! The Row ID (Page# + Slot#) remains identical, meaning zero secondary B+ Tree indexes need to be updated!

3.2 Tuple Header & MVCC Visibility Tags

Every row on disk is prefixed by a mandatory Tuple Header (23+ bytes in PostgreSQL) containing hidden system columns:

  • xmin: The Transaction ID (TXID) that inserted this version.
  • xmax: The Transaction ID (TXID) that deleted/updated this version (0 if alive!).
  • NULL Bitmap: A bit array recording which columns are NULL (saving disk space by storing zero data bytes!).

Module 4: B+ Tree Indexing & Search Mechanics

4.1 Why B+ Trees Beat Binary Search Trees

🛑 Real-World Problem Solved: Deep Disk Seek Latency In a Binary Tree, each node has only 2 children. Indexing 20 million records creates a tree height of 24 levels! Traversing 24 levels requires 24 random disk seeks ($24 \\times 10ms = 240ms$ per lookup!).

A B+ Tree is an N-ary balanced search tree optimized for 8KB disk pages. With a branching factor of $N=500$, a 3-level tree indexes 125,000,000 records! Finding any record takes at most 3 disk seeks!

Furthermore, because the Root and Level 1 nodes are permanently pinned in Buffer Pool RAM, finding a record in 100 million rows requires only 1 physical disk read!

Index Type Leaf Node Contents Pros & Enterprise Use Case
Clustered Index (InnoDB PK) Stores actual physical row data! Blazing fast PK lookups. Only ONE clustered index allowed per table.
Secondary Index (Non-Clustered) Stores indexed key + pointer to Primary Key / RID. Allows indexing secondary attributes (email, status). Requires double lookup!
Covering Index (Index-Only Scan) Includes requested columns in index leaf! ⭐ 10x Speedup! Answers query directly from index without touching table heap!

4.2 Page Splits & Random UUID Thrashing

🛑 Real-World Problem Solved: 50% Index Bloat & I/O Spikes Why do senior architects forbid random UUIDv4 as Primary Keys in MySQL InnoDB? Because random UUIDs insert anywhere across the B+ Tree! This triggers endless 8KB Page Splits, causes 50% index fragmentation bloat, and forces random disk I/O! Always use sequential UUIDv7 or ULID!

Module 5: Concurrency Control & MVCC

5.1 Multi-Version Concurrency Control (MVCC)

The golden rule of modern RDBMS: Readers don't block writers, and writers don't block readers!

When a transaction updates row #10, it does NOT overwrite physical disk memory! It inserts a brand new tuple version with timestamp xmin = current_txid, and marks the old tuple with xmax = current_txid. Concurrent read queries inspect tuple headers and ignore any versions created after their transaction snapshot began!

5.2 Dead Tuples & The Role of VACUUM

🛑 Real-World Problem Solved: Table Bloat & Degradation Because MVCC never overwrites old data in place, updating a row 10,000 times creates 1 alive tuple and 9,999 dead tuples! Without cleanup, table size bloats from 100MB to 50GB!

PostgreSQL VACUUM / Autovacuum: Background worker processes scan pages, remove dead tuple line pointers, and make free space available for new inserts! Never kill or disable Autovacuum!

Module 6: Durability & Crash Recovery (WAL)

6.1 Write-Ahead Logging (WAL) Architecture

🛑 Real-World Problem Solved: Random Disk I/O Saturation Updating 5 scattered rows modifies 5 different 8KB physical disk pages. Writing those 5 random blocks to disk requires 5 slow random disk seeks (~50ms total!). If 1,000 users check out simultaneously, random I/O freezes the server!
✅ Architectural Solution: Converting Random I/O to Sequential I/O The WAL Protocol mandates that before any modified dirty page in RAM is written to disk, the change record MUST be sequentially appended and flushed (fsync) to the WAL log file! Appending bytes to the end of a log file requires zero disk head movement (< 0.1ms per write)!

6.2 ARIES Crash Recovery Algorithm

When a database server reboot occurs after a power outage at 3:00 AM, all RAM buffer pools are erased! The storage engine executes the ARIES protocol in 3 distinct phases:

  1. 1. Analysis Phase: Scans WAL forward from last Checkpoint LSN to identify all Active Transactions & Dirty Pages at crash time.
  2. 2. Redo Phase (Repeating History!): Replays ALL WAL log records forward, restoring disk pages to their exact physical state at crash time!
  3. 3. Undo Phase: Scans WAL backward, executing compensation log records to rollback all uncommitted transactions!

Module 7: Why is the Database Slow? (Diagnosis Framework)

When query P99 latency spikes in production, senior engineers execute a structured elimination framework:

Root Cause Category Key Symptom / Metric Senior Architectural Fix
1. Buffer Pool Thrashing Buffer Pool Hit Ratio < 95%, high disk iowait. Working set exceeds RAM. Increase RAM, partition old archives, or kill queries doing accidental table scans!
2. Index Bloat & Page Splits Table size 5x larger than data; random UUID keys. Switch to sequential UUIDv7/ULID. Execute REINDEX CONCURRENTLY during off-peak hours!
3. Lock Contention & Bloat Queries stuck in waiting for lock; TXID wraparound warning. Kill long-running transactions (> 30 mins) that block Autovacuum! Shard hot inventory counters!
4. Stale Statistics (CBO) CBO chooses Full Table Scan instead of Index Scan. Run ANALYZE table_name; to update histogram distributions in system catalogs!

Module 8: Beyond Relational: LSM & Columnar Storage

8.1 LSM-Trees vs B+ Trees (RocksDB / Cassandra)

For write-heavy IoT event logging (100,000 writes/sec), B+ Tree page splits cause massive write amplification. LSM-Trees (Log-Structured Merge-Trees) turn ALL writes into sequential appends by buffering mutations in a RAM MemTable and flushing them to disk as immutable sorted SSTables!

8.2 Columnar vs Row Storage (OLAP vs OLTP)

In analytical BI reporting (aggregating 1 billion sales rows), row-oriented storage (PostgreSQL) reads all 50 columns from disk into RAM just to access 1 salary integer! Columnar Storage (Parquet, ClickHouse, Snowflake) stores values of a single column contiguously across disk blocks, cutting disk I/O by 90% and enabling SIMD vectorization!

8.3 Bloom Filters in Storage Engines

A Bloom Filter is a space-efficient probabilistic bit array stored in RAM in front of every disk file. In ~10 nanoseconds, it answers either "Definitely NOT present!" (skipping zero-value disk seeks!) or "Possibly present!" (with a tiny 1% false positive rate).
